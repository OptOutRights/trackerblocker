Where third-party requests come from

A web page is not one request. It is more like opening a tiny city: the main document arrives first, then scripts, images, frames, beacons, redirects, workers, headers, and browser features can all start more traffic. TrackerBlocker sits at the browser's request gate and decides what to observe, explain, allow, restrict, or block using packaged local rules.

The Current Gate

Today the extension mostly works at one powerful checkpoint: browser.webRequest.onBeforeRequest. That is early enough to cancel many network requests before they leave the browser. It now pairs that gate with lifecycle, redirect, and request-header hooks, but it still does not explain everything by itself.

1. Page asks A script, image, frame, beacon, XHR, or navigation requests a URL.
2. Browser emits event Firefox exposes request, redirect, completion, error, or header events through webRequest.
3. Classify site The request host is compared with the current page site.
4. Check catalog Known local entries say block, allow, or unknown.
5. Decide User override, site pause, host rules, and path rules produce allow, restrict, or block.

Request Source Map

Each card explains the concept in plain language, then maps it to the inspection surface, interception point, and TrackerBlocker's current behavior. Page inspection can reveal many visible requests, but not every cause: browser-owned traffic, DNS/CNAME details, cache-only service-worker responses, preconnects, and extension-internal messages need other diagnostics. TrackerBlocker now labels those weak spots instead of claiming complete visibility.

Top-Level Page Navigation

User clicks link
New main_frame URL
Reset tab state
Implemented

This is the browser loading a new page in the main tab. Think "I am going to a new address." It is not a tracker by itself, but it tells the extension to forget the old page's request list.

Source

User clicks, redirects, forms, scripts, or session restore.

How to inspect

Mostly yes. The address bar and Network panel show main document loads; session restore or extension causes need browser/extension logs.

How to intercept

onBeforeRequest for main_frame, plus tabs.onUpdated.

Current solution

Resets tab observations and clears pause-once when the site changes.

Parser Subresources

HTML document
script
image
stylesheet
font/media/manifest
Partial

When the browser reads HTML, it automatically discovers more things to fetch. A page can look empty in the address bar while dozens of subresources quietly load behind it.

Source

Tags like <script>, <img>, <link>, video, fonts, embeds, and manifests.

How to inspect

Usually yes. Inspector shows markup and Network shows fetched assets, though cache, CSS imports, and favicon behavior can hide context.

How to intercept

onBeforeRequest, then inspect details.type.

Current solution

Common Firefox request types now map to clear labels, including fonts, media, WebSockets, manifests, objects, and prefetch-like requests.

Remaining fallback: browser-specific unknown types still use other.

Script Fetch And XHR

Page script
fetch('/track')
analytics host
Implemented

JavaScript can send data after the page loads. This is common for analytics events, ad conversions, identity matching, experimentation, and product usage tracking.

Source

fetch(), XMLHttpRequest, SDKs, workers, and third-party scripts.

How to inspect

Yes for requests, not always cause. Network shows Fetch/XHR, but minified SDKs and workers can obscure the initiator.

How to intercept

onBeforeRequest for xmlhttprequest.

Current solution

Mapped to xhr, then blocked only by hostname catalog or user override.

Future: path rules for /collect, /track, /events, /conversion.

Beacon And Ping

Click or page exit
tiny event payload
measurement server
Partial

Beacons are small "just letting you know this happened" messages. They are popular because they can fire when you leave a page, click a button, or scroll to a target.

Source

navigator.sendBeacon(), link pings, unload tracking, analytics SDKs.

How to inspect

Partly. Network may show beacons and pings if logging is open and preserved before clicks or page exit.

How to intercept

onBeforeRequest for beacon and ping where Firefox exposes them.

Current solution

Both map to beacon, but decisions are still hostname-based.

Third-Party Scripts

example.com
tracker-cdn.com/sdk.js
more requests
Implemented

A third-party script is code from another company running on the current page. Once loaded, it can observe page behavior and start more network requests.

Source

Ad tags, analytics SDKs, social widgets, replay tools, payment helpers, fraud tools.

How to inspect

Yes for loaded files. Inspector, Network, and Sources show script URLs; behavior still needs code or request analysis.

How to intercept

onBeforeRequest for script.

Current solution

Known tracker script hosts block. Functional hosts can be cataloged as allow.

Important: no script content inspection today.

Tag Managers

tag-manager.js
ads
analytics
site tools
Partial

A tag manager is a loader for other loaders. Marketing teams use it to add or remove tracking tools without changing site code. Blocking it can help a lot, but can also break things if the site uses it for functional scripts.

Source

Google Tag Manager and similar containers that inject downstream tags.

How to inspect

Partly. DevTools shows the loader and downstream requests, but not always the container rules or exact causality chain.

How to intercept

Block/restrict the loader or block the downstream requests it creates.

Current solution

No special model. Hostname catalog rules only, with no child-request attribution.

Needs metadata and breakage review before aggressive defaults.

Iframes

Page
Embedded mini-page
its own resources
Partial

An iframe is a page inside a page. Ads, videos, comments, payment forms, maps, CAPTCHA, and social widgets often arrive as iframes. Each iframe can have its own scripts and requests.

Source

<iframe>, embedded widgets, ads, maps, players, login, checkout, CAPTCHA.

How to inspect

Usually yes. Inspector shows iframe elements and Network shows frame documents, but cross-origin contents can be restricted.

How to intercept

onBeforeRequest for sub_frame; use documentUrl or originUrl.

Current solution

sub_frame maps to iframe. Frame ancestry is not stored beyond frameId.

Requests Inside Frames

ad iframe
pixel
script
XHR
Partial

The iframe is only the first layer. Once that mini-page loads, it can make its own requests. For a blocker, the hard question is not just "what host?" but "who caused this?"

Source

Scripts, pixels, XHR, images, and nested frames inside a third-party frame.

How to inspect

Partly. Network can show frame requests, but nested frames, redirects, and early requests need stored ancestry for clear explanation.

How to intercept

Use webRequest and preserve top-level site plus frame initiator.

Current solution

Aggregates by request host. No "loaded by this frame" explanation yet.

Redirect Chains

click
ad.net
measure.io
final site
Gap

Redirects are handoffs. A request goes to one URL, which tells the browser to go somewhere else, then somewhere else again. Ads, affiliate links, login, and measurement flows use this a lot.

Source

Ads, affiliate links, login flows, consent tools, and measurement URLs.

How to inspect

Yes when captured. Preserve Network logs before navigating; fast hops, blocked requests, and downloads may still need redirect events.

How to intercept

Add onBeforeRedirect, plus onCompleted and onErrorOccurred.

Current solution

Stores visible redirect hops as source host, destination host, status code, and timestamp.

Full redirect URLs and query strings are not stored.

WebSockets

Page
wss:// live pipe
server stays connected
Partial

A WebSocket is a long-lived two-way connection. Instead of many small request/response trips, the page opens a pipe and keeps it open. Chat, dashboards, tracking, and personalization can use it.

Source

ws: and wss: connections opened by pages and SDKs.

How to inspect

Yes while open. Network can show WebSocket handshakes and frames, but closed or worker-originated sockets are harder to attribute later.

How to intercept

onBeforeRequest where Firefox exposes WebSocket handshakes.

Current solution

Site classification supports ws/wss, and exposed WebSocket handshakes map to websocket.

EventSource And Streaming

Page opens stream
server keeps sending
generic request type
Partial

Some requests stay open so the server can keep pushing updates. They may look like normal XHR/fetch at the browser hook even though the behavior is closer to a live feed.

Source

Server-sent events, streaming fetch, long polling, chat, notifications, experiments.

How to inspect

Yes for visible connections. Network shows long-lived requests, but labels may look like ordinary XHR/fetch without more context.

How to intercept

onBeforeRequest; improve labels with path and response metadata later.

Current solution

Covered as ordinary network requests, with no separate streaming category.

Workers And Worklets

Main page
background helper thread
its own requests
Partial

Workers are JavaScript helpers that run off the main page thread. They can import scripts and make requests without being directly tied to a visible button or image.

Source

Dedicated workers, shared workers, service workers, audio worklets, paint worklets.

How to inspect

Partly. DevTools can inspect some workers and requests; worklets and initiators are not always obvious from the DOM.

How to intercept

Use webRequest for worker script loads and worker-originated network requests.

Current solution

Covered only when Firefox exposes normal tab-associated request events.

Future: store worker/initiator details when available.

Service Workers

Page asks
service worker intercepts
cache or network
Gap

A service worker is a site's local proxy. It can answer from cache before the network is touched. That means "I did not see a network request" is not always the same as "nothing happened."

Source

Offline caches, app shells, background sync-like behavior, fetch handlers.

How to inspect

Partly, not completely. DevTools shows registrations and caches, but cache-served responses may not appear as network rows.

How to intercept

Use webRequest only for requests that actually hit the network.

Current solution

No special handling. Cache-served responses may not be visible.

Preload, Prefetch, Preconnect

Page hints
preload
DNS
socket warmup
Partial

Pages can ask the browser to prepare things before they are needed. Some hints fetch real URLs; others only resolve DNS or open a connection, which may not look like a normal request.

Source

preload, prefetch, prerender, DNS prefetch, and preconnect hints.

How to inspect

Partly. Inspector shows hints and Network shows URL fetches; DNS prefetch and preconnect may not appear as requests.

How to intercept

Use webRequest for URL-bearing fetches. DNS and connection setup may be weakly visible.

Current solution

URL-bearing requests may be seen. No distinct prefetch/preconnect category.

Response Headers That Spawn Work

Server response
Link / policy / redirect
browser fetches more
Gap

Not every request begins in the visible HTML. A server response can include headers that tell the browser to fetch, report, redirect, cache, or enforce policies.

Source

Link headers, redirects, refresh behavior, reporting endpoints, policy headers.

How to inspect

Partly. Network shows response headers and follow-on requests; DOM inspection alone will not reveal header-driven causes.

How to intercept

Use onHeadersReceived if the extension needs to inspect response headers.

Current solution

Only the resulting requests are observed. Headers are not inspected.

Reporting, CSP, NEL

Browser notices event
report upload
report collector
Partial

Sites can configure the browser to send reports about security violations, network failures, and errors. These can be useful, but they are still data leaving the browser.

Source

CSP reports, Network Error Logging, Reporting API, crash/error SDK endpoints.

How to inspect

Partly. Headers and Network uploads may reveal reports, but they often fire only after errors, violations, or navigation.

How to intercept

Use webRequest for visible report uploads; add path/type labels later.

Current solution

Covered only as ordinary requests if visible. No reporting category.

Outgoing Headers

Request URL
cookies + referrer
identifier travels
Gap

The URL is only part of the story. Requests can carry cookies, referrers, user-agent hints, authorization data, and cached identifiers. A "restrict" mode would target this layer.

Source

Cookies, referrers, authorization headers, user-agent hints, ETags, cache validators.

How to inspect

Yes for captured requests. Network shows request headers and cookies, but sensitive fields should stay out of default product behavior.

How to intercept

onBeforeSendHeaders, cookie APIs, and careful header modification.

Current solution

Restrict mode removes outgoing Cookie and Referer headers for reviewed third parties.

Header values are not stored; response headers are not modified.

Incoming Headers

Third-party response
Set-Cookie / cache state
browser stores ID
Gap

A third party can set state when it responds. That state can help recognize the browser later. Blocking the request prevents this; restricting would require header-level control.

Source

Set-Cookie, redirects, cache headers, HSTS, reporting headers, browser state.

How to inspect

Yes for captured responses, partly for stored state. Network shows headers; Storage shows some resulting state.

How to intercept

onHeadersReceived, used conservatively because it can break sites.

Current solution

Not implemented. Response headers are not inspected or modified.

Web Storage

Third-party script
localStorage
IndexedDB
Cache API
Gap

Tracking is not only network. Once a third-party script runs, it may write identifiers into browser storage for its origin. This is hard to intercept cleanly from a WebExtension.

Source

localStorage, IndexedDB, Cache API, service worker caches, other origin storage.

How to inspect

Partly. Storage shows values by origin, but usually not which script wrote them or why.

How to intercept

Content-script API patching is possible but fragile. Network blocking is cleaner.

Current solution

Not inspected. TrackerBlocker only uses its own browser.storage.local.

CNAME Cloaking

metrics.example.com
DNS alias
tracker vendor
Gap

CNAME cloaking makes a tracker look first-party. The URL says metrics.example.com, but DNS points it to a third-party tracking service behind the scenes.

Source

First-party-looking hostnames delegated to third-party tracking infrastructure.

How to inspect

No, not from page inspection alone. DevTools shows the hostname, not the DNS CNAME chain behind it.

How to intercept

Need DNS/CNAME knowledge, curated rules, or browser-provided canonical names.

Current solution

Not implemented. Same eTLD+1 requests are treated as first-party.

Important gap for mature blocking, but not trivial offline.

Mixed-Use Infrastructure

Cloud/CDN/payment
site needs it
may also measure
Partial

Some third parties are both useful and privacy-relevant. Payments, CAPTCHA, CDNs, maps, video, login, and security tools can be necessary for the page to work.

Source

CDNs, cloud providers, CAPTCHA, payments, login, maps, video, security vendors.

How to inspect

Partly. DevTools shows hosts, paths, frames, and headers; product meaning needs catalog context and breakage testing.

How to intercept

Prefer allow or restrict. Use narrow host/path rules for confirmed tracking endpoints.

Current solution

Catalog can mark services as allow, block, or restrict, and precise path rules can override broad host defaults.

Internal URLs

about:
moz-extension:
data:
blob:
Implemented

Not every URL is a website. Browser pages, extension pages, inline data, blobs, and local files are different from normal HTTP traffic and should not be called third-party trackers.

Source

about:, moz-extension:, data:, blob:, file:, and related schemes.

How to inspect

Yes for visible references, but they are not normal third-party web requests. Blob/data URLs can be short-lived.

How to intercept

Classify non-web schemes as ignored or unclassifiable.

Current solution

Only http, https, ws, and wss are comparable web schemes.

Browser-Managed Requests

Firefox or extension
no content tab
ignore for page UI
Implemented

Some requests belong to the browser, another extension, update checks, safe browsing, or background contexts. Mixing these with page requests would make the popup confusing.

Source

Browser services, other extensions, updates, safe browsing, captive portal checks.

How to inspect

No, not from a page. These belong to browser or extension debugging contexts and should stay outside page summaries.

How to intercept

Keep browser-owned traffic separate and label it if ever shown.

Current solution

Requests with tabId < 0 are intentionally skipped.

Downloads And Protocols

Click
download or app handoff
maybe tracked redirect
Partial

Some clicks leave normal page loading and become downloads or external app handoffs. Ad and affiliate flows can use redirects before that happens.

Source

Downloads, app links, protocol handlers, affiliate redirects.

How to inspect

Partly. Network may show pre-handoff redirects; after Downloads or external apps take over, page inspection is not enough.

webRequest before handoff; downloads API only if future diagnostics need it.

Form Posts

User submits form
POST request
first or third party
Partial

A form can submit data to the current site or to another service. For privacy, the blocker should usually classify the destination host without reading the form body.

Source

Newsletter forms, login, checkout, marketing forms, embedded services.

How to inspect

Yes for destination and headers. Inspector shows form attributes and Network shows submissions; bodies need privacy caution.

How to intercept

onBeforeRequest for navigation or subresource form submissions.

Current solution

Covered as ordinary visible requests. Request bodies are not inspected.

Extension UI Messages

Popup
runtime message
background state
Implemented

The popup and options pages talk to the background script. This is not web tracking traffic, but it is part of the system that shows summaries and updates user rules.

Source

Popup summary requests, settings reads, site pause, hostname override, reset.

How to inspect

No, not from the web page. These are extension-internal messages visible through extension debugging or logs.

How to intercept

browser.runtime.onMessage inside the extension.

Current solution

Implemented and kept separate from page request observation.

Implemented Gap Closers

TrackerBlocker now deepens the current request gate in local-only layers. The remaining gaps are labeled where Firefox or WebExtensions cannot provide complete page-level visibility.

Layer Current behavior Privacy boundary
Catalog metadata Entries can carry specific source/confidence provenance, breakage risk, notes, and precise path rules without presenting generic review labels as evidence. Metadata is packaged JSON only; no remote classification.
Path-level rules Host defaults can be overridden by local path-prefix or url-prefix rules. Rules match URL path/prefix only; request bodies are not inspected.
Redirect recording Visible redirect hops record source host, destination host, status code, and timestamp. Full redirect URLs and query strings are not stored.
Restrict mode Reviewed third parties can load while outgoing Cookie and Referer headers are removed. Header values are not stored, and response headers are not modified.
Frame/initiator ancestry Rows show browser-exposed frame ids, parent frame ids, document hosts, and initiator hosts. This is not full script-level causality and is labeled as limited.

Mental Model

Page layer

HTML, scripts, frames, workers, forms, storage APIs, and user interactions create demand.

Browser layer

Firefox turns that demand into network requests, cache hits, DNS, connections, and reports.

Extension layer

TrackerBlocker observes request events, compares sites, checks local rules, and returns allow or cancel.

User layer

The popup explains what was seen and lets the user pause, allow, or block a hostname.

This HTML guide is a learning companion to docs/reasoning/third-party-request-sources.md. It is not a new blocking policy by itself. The core rule still applies: keep tracker intelligence local, packaged, privacy-preserving, and cautious about mixed-use services.
